Goal: Harden gh-sweep and turn it into a practical GitHub Actions storage cleanup TUI/CLI.

Repository context
==================

Local repository:

  C:\dev\gh-sweep

Fork remote:

  origin   https://github.com/andreamancuso/gh-sweep.git
  upstream https://github.com/KyleKing/gh-sweep.git

This repository is a Go/Bubble Tea GitHub repository-management TUI. It already has useful structure: Cobra commands, GitHub API helpers, Bubble Tea components, config/cache modules, branch cleanup features, Actions analytics, and release overview. It does not currently solve the specific problem that prompted this work: inspecting and reclaiming GitHub Actions / Packages storage from a local terminal UI without relying on GitHub Actions itself.

The intended outcome is twofold:

1. Fix the security issues found during review so the tool is safe to run with GitHub write/delete scopes.
2. Add a storage cleanup feature set focused on GitHub Actions artifacts, caches, workflow run logs, releases/assets, and packages.

1. Replace github.com/cli/go-gh v1
----------------------------------

govulncheck reports reachable vulnerability GO-2024-3295:

  Violation of GitHub host security boundary when sourcing authentication token within a codespace.

Affected module:

  github.com/cli/go-gh@v1.2.1

Relevant local code:

  internal/github/client.go

The current client imports:

  github.com/cli/go-gh
  github.com/cli/go-gh/pkg/api

and uses:

  gh.RESTClient(opts)
  gh.HTTPClient(opts)
  api.ClientOptions
  api.RESTClient

The Go vulnerability database indicates github.com/cli/go-gh v1 has no known fixed version, while github.com/cli/go-gh/v2 has a fix path.

Implementation requirement:

  Migrate from github.com/cli/go-gh v1 to github.com/cli/go-gh/v2 if feasible.

Alternative if v2 migration is too disruptive:

  Remove go-gh from the auth-critical path and use a narrower custom client:

    - Read token from gh auth token via explicit subprocess only when requested, or
    - Read token from GITHUB_TOKEN/GH_TOKEN, or
    - Support explicit token injection for tests.

Do not silently mix hosts/tokens. If the repo is on github.com, use a github.com token. If enterprise host support is added later, host selection must be explicit and tested.

Acceptance criteria:

  - govulncheck no longer reports GO-2024-3295 as reachable.
  - GitHub API requests still work with normal gh auth.
  - Tests pass.
  - Auth behavior is documented.

3. Remove or lock down plaintext token config
---------------------------------------------

Current config model includes:

  internal/config/config.go

  type GitHubConfig struct {
      Token  string `yaml:"token"`
      APIURL string `yaml:"api_url"`
  }

The config save path writes YAML with:

  os.WriteFile(path, data, 0644)

Risks:

  - A GitHub token may be stored in plaintext.
  - On Unix-like systems, mode 0644 is world-readable by default.
  - On Windows, mode handling differs, but persisting tokens in a regular YAML config is still poor default design.
  - NewClientWithToken exists but appears unused; token config may be incomplete/dead code.

Implementation requirement:

  Prefer removing or deprecating github.token from persistent config.

Recommended auth model:

  - Primary: reuse gh CLI auth.
  - Secondary: GH_TOKEN or GITHUB_TOKEN environment variable.
  - Testing: explicit token only through NewClientWithToken or equivalent test seam.
  - Do not save tokens to .gh-sweep.yaml.

If token persistence remains:

  - Save config with 0600 on Unix-like systems.
  - Save config directories with 0700.
  - Redact token fields from any rendered config/debug output.
  - Document that config-token storage is discouraged.

Acceptance criteria:

  - No default path writes GitHub tokens to disk.
  - Config save uses private permissions for app-private files.
  - README no longer recommends putting a long-lived token in YAML.


4. Gate destructive CLI operations
----------------------------------

The TUI orphan-branch cleanup has a y/n confirmation. The CLI cleanup path does not.

Current risky path:

  gh-sweep orphans --cleanup

This deletes all detected orphan branches immediately unless --dry-run is provided.

Relevant code:

  cmd/orphans.go
  internal/tui/components/orphans/model.go

Implementation requirement:

  Make destructive CLI operations safe by default.

Preferred behavior:

  - List/preview by default.
  - --cleanup without --yes should print the target count and require confirmation.
  - --yes should be required for non-interactive deletion.
  - --dry-run should remain available and should never mutate.
  - For high-impact operations, typed confirmation is better than single-character confirmation.

Recommended confirmation text:

  To delete N branches from namespace OWNER_OR_ORG, type: OWNER_OR_ORG

Acceptance criteria:

  - gh-sweep orphans --cleanup no longer deletes without confirmation.
  - gh-sweep orphans --cleanup --dry-run deletes nothing.
  - gh-sweep orphans --cleanup --yes performs deletion non-interactively.
  - Tests cover dry-run, confirmation-required, and --yes paths.


5. Escape GitHub REST path segments
-----------------------------------

Several GitHub API paths are built with fmt.Sprintf and raw path segments:

  internal/github/branches.go
  internal/github/collaborators.go
  internal/github/subscriptions.go

Examples:

  repos/%s/%s/git/refs/heads/%s
  repos/%s/%s/collaborators/%s
  repos/%s/%s/subscription

Risks:

  - Branch names commonly contain slash characters.
  - Other path segment characters can produce incorrect endpoint paths.
  - This is not necessarily credential exposure, but it is correctness-sensitive and can become a security issue around destructive endpoints.

Implementation requirement:

  Centralize GitHub API path construction.

Recommended helper:

  func pathJoin(parts ...string) string

Each dynamic path segment should be URL path-escaped. Be careful with branch refs: GitHub's "refs/heads/{branch}" style endpoints may need branch names encoded as a single path tail or use APIs that accept the ref in a query-safe/body-safe way. Test branch names containing "/" explicitly.

Acceptance criteria:

  - Branch delete works for branch names like feature/foo.
  - Compare works for branch names like feature/foo and bugfix/bar.
  - Collaborator/subscription methods keep current behavior.
  - Tests cover path construction.

Relevant files:

Acceptance criteria:

  - explicit export-file behavior is intentional and tested or documented.


7. Review subprocess usage
--------------------------

  internal/git/local.go

Because the code uses exec.Command with argument slices and does not invoke a shell, this is not classic shell injection. Still, Git ref arguments should be handled defensively.

  - Check parsing errors in CompareBranches instead of ignoring fmt.Sscanf errors.

Acceptance criteria:

  - gosec G204 findings are either resolved or intentionally suppressed with comments explaining why exec.Command is safe here.
  - CompareBranches returns an error when ahead/behind parsing fails.

Storage cleanup feature: desired product behavior
=================================================

Add a GitHub storage cleanup feature to gh-sweep. This is the original user problem.

The user should be able to run:

  gh-sweep storage --repo owner/repo

or launch it from the full TUI.

The feature must work locally. It must not depend on GitHub Actions running, because the failure mode is often "Actions are blocked because storage quota is exceeded."


Storage dashboard
-----------------

For one repository, show:

  - Actions artifacts:
      count
      total size
      largest artifacts
      age
      name
      expired state

  - Actions caches:
      count
      total size
      key
      ref
      created_at
      last_accessed_at

  - Workflow runs:
      counts grouped by workflow and conclusion
      oldest/newest
      failed/cancelled count
      successful release runs count

  - Releases/assets:
      release count
      asset count
      total asset size
      per-release total size

  - Packages:
      package type
      package name
      package versions
      approximate metadata available from API
      note that exact storage accounting may be limited by GitHub API

  - Repo git size:
      use repository diskUsage from GraphQL/gh API if available

Important:

  The dashboard must distinguish source repo git size from Actions/Packages storage. In the bt-browser case, repo git size was about 7 MB while Actions artifacts were about 1.5 GiB.


Storage commands
----------------

Recommended command structure:

  gh-sweep storage --repo owner/repo
  gh-sweep storage --repo owner/repo --list
  gh-sweep storage --repo owner/repo --delete-artifacts --older-than 3d
  gh-sweep storage --repo owner/repo --delete-all-artifacts
  gh-sweep storage --repo owner/repo --delete-caches
  gh-sweep storage --repo owner/repo --delete-runs --conclusion failure,cancelled
  gh-sweep storage --repo owner/repo --delete-runs --older-than 30d
  gh-sweep storage --repo owner/repo --inspect-releases
  gh-sweep storage --repo owner/repo --inspect-packages

All destructive flags should be dry-run by default unless --yes is supplied or an interactive confirmation succeeds.


Storage TUI interactions
------------------------

Recommended TUI layout:

  owner/repo

  Artifacts      114      1.50 GiB
  Caches         0        0 B
  Runs           80       logs retained
  Releases       3        206 MiB
  Packages       0        0 B / unknown
  Repo Git       7 MiB

  [1] Artifacts
  [2] Caches
  [3] Workflow runs
  [4] Releases
  [5] Packages

  D dry-run selected cleanup
  X execute selected cleanup
  R refresh
  Q quit

Rows should be sortable by size, date, type, workflow, and conclusion where relevant.


Artifact cleanup
----------------

GitHub REST endpoints:

  GET    /repos/{owner}/{repo}/actions/artifacts
  DELETE /repos/{owner}/{repo}/actions/artifacts/{artifact_id}

Behavior:

  - List all artifacts with pagination.
  - Compute total size.
  - Show largest artifacts first.
  - Support deletion by selected rows, by age, or all.
  - Confirm exact count and approximate size before deletion.
  - Treat deletion as irreversible.

Acceptance criteria:

  - Can reproduce the manual cleanup previously done with gh:

      deleted 114 artifacts
      reclaimed about 1.50 GiB

  - API verification shows artifact count 0 after deletion.


Actions cache cleanup
---------------------

GitHub REST endpoints:

  GET    /repos/{owner}/{repo}/actions/caches
  DELETE /repos/{owner}/{repo}/actions/caches/{cache_id}

Behavior:

  - List caches with pagination if needed.
  - Show key, ref, size, created_at, last_accessed_at.
  - Support deleting selected/all/older-than.

Acceptance criteria:

  - Empty cache list displays cleanly.
  - Deleting selected caches verifies count/size afterward.


Workflow run cleanup
--------------------

GitHub REST endpoints:

  GET    /repos/{owner}/{repo}/actions/runs
  DELETE /repos/{owner}/{repo}/actions/runs/{run_id}

Behavior:

  - List runs with pagination.
  - Group by workflow name and conclusion.
  - Support deleting failed/cancelled runs.
  - Support deleting runs older than N days.
  - Allow preserving successful Release runs by default.
  - Explain that deleting a workflow run also removes artifacts associated with that run.

Safe defaults:

  - Never delete successful release runs by default.
  - Prefer deleting failed/cancelled CI runs first.
  - Require confirmation for any successful run deletion.

Acceptance criteria:

  - Can reproduce the previous manual cleanup:

      delete 27 failed/cancelled CI runs
      keep successful CI and all Release runs

  - Post-cleanup grouping shows only expected run classes.


Release asset inspection
------------------------

GitHub REST endpoints:

  GET /repos/{owner}/{repo}/releases

Behavior:

  - List releases and asset sizes.
  - Summarize total release-asset size.
  - Deletion should be opt-in and conservative.

Safe defaults:

  - Do not delete releases or release assets in "recommended cleanup."
  - Require explicit selection and typed confirmation for release deletion.

Acceptance criteria:

  - For bt-browser-like repos, the tool can report:

      3 releases
      about 206 MiB total release assets

  - It does not delete releases unless explicitly requested.


Package inspection and cleanup
------------------------------

GitHub Packages APIs require package scopes:

  read:packages
  delete:packages

Behavior:

  - Detect missing scope and show the exact gh command needed:

      gh auth refresh -s read:packages -s delete:packages

  - Inspect package types:

      container
      nuget
      npm
      maven
      rubygems

  - Filter packages linked to the target repository where GitHub exposes repository metadata.
  - Deletion must be explicit and typed-confirmed.

Acceptance criteria:

  - Missing package scopes are handled as a non-fatal diagnostic.
  - Empty package inventory displays cleanly.


Recommended cleanup policy
==========================

Provide a "recommended cleanup" action that:

  - Deletes Actions artifacts older than 3 days.
  - Deletes failed/cancelled CI runs.
  - Deletes Actions caches older than a configurable threshold.
  - Preserves releases.
  - Preserves successful release runs.
  - Preserves source branches/tags.
  - Does not rewrite git history.

All recommended cleanup must be preview-first:

  - Show target counts.
  - Show estimated reclaimed size where known.
  - Show exact categories to be affected.
  - Require confirmation or --yes.


Safety requirements for the cleanup feature
===========================================

Non-negotiable:

  - Dry-run/preview first for destructive operations.
  - --yes required for non-interactive deletion.
  - Typed confirmation for releases, packages, successful workflow runs, or broad multi-repo deletion.
  - Never rewrite git history.
  - Never delete branches/tags as part of storage cleanup.
  - Never print tokens.
  - Never store tokens in exported reports.
  - Rate-limit and pagination handling must be explicit.
  - Partial failures must be reported item-by-item.
  - After deletion, refresh and verify remaining counts.


Documentation updates

Update README with:

  - Installation from fork/source.
  - Auth requirements and scopes.
  - Security posture: local tool, uses gh auth/env token, no token persistence by default.
  - Storage cleanup examples.
  - Explanation of GitHub storage categories:

      repo git storage

Definition of done
==================

The work is done when:

  - The tool has no reachable govulncheck findings.
  - Tests, vet, and build pass.
  - Destructive operations are safe by default.
  - Tokens are not persisted by default.
  - The storage cleanup feature can list artifacts, caches, runs, releases, and packages.
  - The storage cleanup feature can delete artifacts, caches, and failed/cancelled workflow runs with preview/confirmation.
  - The storage cleanup feature preserves releases and successful release runs unless explicitly selected.
  - Documentation explains scopes, risks, and examples.
